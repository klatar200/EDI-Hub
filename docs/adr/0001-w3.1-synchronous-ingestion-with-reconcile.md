# ADR 0001: W3.1 — Synchronous ingestion parse + startup reconcile (no BullMQ for v1)

**Status:** Accepted  
**Date:** 2026-06-25  
**Deciders:** Keagan (product/engineering)  
**Build plan item:** [BUILD_PLAN.md §8 W3.1](../../BUILD_PLAN.md#81--async-queue-choose-one-write-adr)

---

## Context

The original build plan assumed **BullMQ + Redis** for async ingestion: HTTP returns quickly, a worker parses from S3, retries are free. That matches a high-volume multi-worker SaaS topology.

The codebase evolved differently:

1. **Ingestion parses inline.** After the raw file row is written, `ingestRawFile` calls `parseAndStore` synchronously when status is `RECEIVED` (`apps/api/src/services/ingestion.ts`).
2. **Crash safety.** If the process dies between the DB insert and parse completion, rows stay at `RECEIVED`. On boot, `reconcileStuckReceived` re-parses anything older than 15 minutes (`apps/api/src/index.ts`, `apps/api/src/services/parsing.ts`). This is **F6 / PS-5 Option B**.
3. **Background jobs use Postgres, not Redis.** Detection passes and email digests run through the DB-backed job adapter (`apps/api/src/jobs/db-adapter.ts`, `JOB_BACKEND=db`). Desktop explicitly avoids a Redis dependency.
4. **Operational simplicity for M5.** Staging deploy targets ECS + RDS + S3. Adding ElastiCache is another moving part before first external customer.

We must choose explicitly: adopt BullMQ now, or document sync + reconcile as the v1 architecture and defer Redis.

---

## Decision

**Adopt Option B for v1, staging, and M5:** keep **synchronous inline parse** on the ingestion path and **startup reconcile** for stuck `RECEIVED` rows. **Do not add BullMQ/Redis for ingestion or scheduling in the M5 deploy.**

Background work (detection, email digest) continues on the **Postgres `Job` table** poller already shipped in D2.

---

## Rationale

| Factor | Sync + reconcile | BullMQ + Redis |
|--------|------------------|----------------|
| Matches current code | Yes — already production path | Requires new worker process, queue wiring, ops |
| Desktop / LAN SKU | No Redis required | Redis or second queue backend to maintain |
| Ingest latency | Parse time added to upload response | Fast 202-style ack possible |
| Failure modes | Reconcile + manual reparse (F54) | Queue retry, dead-letter |
| M5 deploy surface | RDS + S3 + ECS only | + ElastiCache, worker scaling |
| Volume fit (v1) | Pilot / low tens of files per minute | Needed at sustained high parallel ingest |

For an observability hub that receives **passive copies** (not live path), parse time on typical 850/856 files is acceptable. Ops already has **retry parse** from triage and lifecycle expand (F54).

---

## Consequences

### Positive

- Docs match running code — no “planned BullMQ” drift.
- One fewer infrastructure dependency for staging and first customer.
- Desktop and SaaS share the same ingestion semantics.
- `reconcileStuckReceived` + `reparseRawFile` provide a clear recovery story without a queue.

### Negative / tradeoffs

- Large files or slow parsers block the ingestion HTTP handler (mitigated: SFTP/AS2 channels are already async from the partner’s perspective).
- Horizontal scale-out does not isolate parse CPU — all work runs in API tasks. Acceptable until ingest volume proves otherwise.
- No built-in queue depth metrics — use `/internal/metrics` + ingest latency logs instead.

### Revisit BullMQ when

Add **ADR 000X** (superseding this one) if **any** of:

- Sustained ingest exceeds what a single API task can parse without SLA breach (measure p95 upload-to-PARSED latency in staging).
- We need **sub-minute** scheduled jobs with high fan-out beyond the DB poller’s comfort zone.
- A dedicated parse-worker tier is required for CPU isolation (e.g. 10k+ files/day per tenant).

The `JobQueue` interface and `JOB_BACKEND` env switch in `apps/api/src/jobs/factory.ts` remain the extension point — implement `bullmq-adapter.ts` only when triggered by the criteria above.

## Implementation notes (already shipped)

| Component | Location |
|-----------|----------|
| Inline parse after store | `apps/api/src/services/ingestion.ts` |
| Startup reconcile | `apps/api/src/index.ts` → `reconcileStuckReceived()` |
| Reconcile threshold (15 min) | `apps/api/src/services/parsing.ts` `DEFAULT_RECONCILE_MINUTES` |
| Manual reparse | `POST /api/raw-files/:id/reparse` |
| DB job queue | `apps/api/src/jobs/db-adapter.ts` |

**Exit criterion (W3.1):** After API restart, no `RECEIVED` row older than reconcile window remains unparsed unless actively failing parse — covered by startup reconcile + ops reparse.
