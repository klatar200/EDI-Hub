# ADR 0002: W3.2 — Same-origin web + API default; CORS allowlist escape hatch

**Status:** Accepted  
**Date:** 2026-06-25  
**Deciders:** Keagan (product/engineering)  
**Build plan item:** [BUILD_PLAN.md §3 W3.2](../../BUILD_PLAN.md#3-architecture-constraints-do-not-reopen)

---

## Context

The web app calls the API with `Authorization: Bearer <Clerk JWT>`. Browsers enforce the same-origin policy on cross-origin requests; credentialed fetches require explicit CORS headers and cannot use `Access-Control-Allow-Origin: *`.

Three deployment shapes exist in this repo:

| Shape | How it works today |
|-------|-------------------|
| **Local dev (SaaS)** | Vite on `:5173` proxies `/api` → `:3000` — browser sees same origin (`apps/web/vite.config.ts`). |
| **Desktop / LAN server** | API serves React static files from `WEB_STATIC_DIR` on one port — true same-origin (`apps/api/src/server.ts`). |
| **Desktop dev (optional)** | Renderer on `:5173`, API on `:3000` — `CORS_ALLOWED_ORIGINS` enables `@fastify/cors` (`apps/api/src/config.ts`). |

Terraform today provisions **`api.<domain>`** on the ALB (`infra/alb.tf`). The API registers authenticated routes under **`/api/*`** and can serve the SPA from **`/`** when `WEB_STATIC_DIR` is set.

We must decide the **default staging/production topology**: single origin (web + API co-located) vs split origins (`app.` + `api.`) with CORS.

---

## Decision

**Default for staging, M5, and first production deploy: same-origin.**

- Build the React app (`npm run build --workspace=@edi/web`).
- Set **`WEB_STATIC_DIR`** to the built `apps/web/dist` in the API container/task.
- Serve everything from **one public hostname** (e.g. `https://app.staging.edihub.example.com` or reuse `api.` if DNS is already wired — the important part is one origin for browser + API).
- Leave **`CORS_ALLOWED_ORIGINS` unset** (empty) so the CORS plugin is **not registered** — closed by default.

**Escape hatch: split-origin with CORS allowlist** when deliberately deploying web and API on different hostnames (e.g. CloudFront for static assets, ALB for API only):

- Set `CORS_ALLOWED_ORIGINS=https://app.staging.edihub.example.com` (comma-separated for Clerk preview URLs / LAN dev origins).
- Set `VITE_API_URL=https://api.staging.edihub.example.com/api` at **web build time** so the SPA calls the remote API.
- Register Clerk authorized origins for **both** hostnames.

---

## Rationale

| Factor | Same-origin (chosen default) | Split-origin + CORS |
|--------|------------------------------|---------------------|
| Security | No cross-origin attack surface; no CORS misconfiguration risk | Requires correct allowlist maintenance |
| Clerk setup | One authorized origin | Multiple origins + redirect URLs |
| Matches desktop LAN | Identical mental model | Different from desktop |
| Infra complexity | API task serves static files (already implemented) | CloudFront + S3 or second deploy target |
| Local dev | Vite proxy already same-origin | N/A |

The codebase already invested in **D4 Sprint 2** (`/api` prefix + optional static serving). Using it for staging avoids new Terraform before M5 proof.

Split-origin remains valid for a future CDN-fronted marketing site or strict API isolation — use the escape hatch, not the default.

---

## Consequences

### Positive

- Staging deploy checklist is simple: one ALB target group, one TLS cert hostname, one Clerk origin.
- No `@fastify/cors` in the default path — preflight never hits auth middleware incorrectly.
- Desktop, dev proxy, and staging share one API contract: browser calls **`/api/...`** relative to the page origin.

### Negative / tradeoffs

- API tasks serve static assets — slightly larger container, cache headers for `/assets/*` should be set appropriately (Vite build uses hashed filenames; long cache is safe).
- Independent CDN cache invalidation for web-only deploys requires either redeploying the API task or moving to split-origin later.

### Staging checklist (operator)

1. Build web: `npm run build --workspace=@edi/web`
2. API env: `WEB_STATIC_DIR=/app/apps/web/dist` (or copy dist into image at build time)
3. API env: `CORS_ALLOWED_ORIGINS=` (empty)
4. Clerk: authorized origin = public URL (e.g. `https://app.staging.edihub.example.com`)
5. Smoke: sign in, upload test EDI, lifecycle page loads without CORS errors in browser devtools

### Split-origin checklist (when needed)

1. Deploy API at `api.<domain>` without `WEB_STATIC_DIR`
2. Deploy web to CDN or static host at `app.<domain>` with `VITE_API_URL=https://api.<domain>/api`
3. `CORS_ALLOWED_ORIGINS=https://app.<domain>` (+ any LAN/dev origins)
4. Clerk: both origins authorized

## Implementation notes (already shipped)

| Component | Location |
|-----------|----------|
| `/api` route prefix | `apps/api/src/server.ts` |
| Static + SPA fallback | `WEB_STATIC_DIR` + `@fastify/static` |
| CORS (opt-in) | `CORS_ALLOWED_ORIGINS` → `@fastify/cors` |
| Web API base URL | `apps/web/src/lib/api.ts` — defaults to `/api` |
| Desktop LAN | Same-origin via `WEB_STATIC_DIR` in desktop main process |
